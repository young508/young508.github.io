View的创建

inflater

Activity的setContentView来设置View界面，最终会调用LayoutInflater.from(mContext).inflate(resId, contentParent);

  • 参数意义

    • inflater方法有三个参数，第一参数是布局文件，第二个参数表示将该布局添加到哪个父View中，第三个参数表示是否允许将该布局添加到该View中
    • 如果root = null,attachToRoot无论为何值，都不会将该View添加到任意父View中
    • 如果root = view1,attachToRoot=true,表示该view添加到view1中，会被view1约束
    • 如果root = view1,attachToRoot=false，表示该view不添加到view1中，但仍然会被view1约束，比如该view的宽，高此时才会按照布局生效。

  • 流程

    • 我们以Activity setContentView为例子，进行如下逻辑步骤

    • 创建xml文件的根节点view，final View temp = createViewFromTag(root, name, inflaterContext, attrs);

    • 最后判断参数，是否添加到root上 root.addView(temp, params);

View的绘制

  • 实际上，View的绘制是由ViewRoot来负责的。每个应用程序窗口的decorView都有一个与之关联的ViewRoot对象，这种关联关系是由WindowManager来维护的。
  • 那么decorView与ViewRoot的关联关系是在什么时候建立的呢？答案是Activity启动时，ActivityThread.handleResumeActivity()方法中建立了它们两者的关联关系。
  • 当建立好了decorView与ViewRoot的关联后，ViewRoot类的requestLayout()方法会被调用，以完成应用程序用户界面的初次布局。实际被调用的是ViewRootImpl类的requestLayout()方法

三个阶段

View的整个绘制流程可以分为以下三个阶段：

  • measure: 判断是否需要重新计算View的大小，需要的话则计算；
  • layout: 判断是否需要重新计算View的位置，需要的话则计算；
  • draw: 判断是否需要重新绘制View，需要的话则重绘制。

performTraversals

Measure

测量，顾名思义，就是计算出view和viewGroup真实的大小，这里面涉及到两个很重要的类MeasureSpec和ViewGroup.LayoutParams

MeasureSpec

  • MeasureSpec概括了从父布局传递给子view布局要求。每一个MeasureSpec代表了宽度或者高度要求，它由size（尺寸）和mode（模式）组成。
  • 有三种可能的mode：UNSPECIFIED、EXACTLY、AT_MOST
    • UNSPECIFIED：未指定尺寸模式。父布局没有对子view强加任何限制。它可以是任意想要的尺寸。
    • EXACTLY：精确值模式。父布局决定了子view的准确尺寸。子view无论想设置多大的值，都将限定在那个边界内。也就是layout_width属性和layout_height属性为具体的数值，如50dp，或者设置为match_parent，设置为match_parent时也就明确为和父布局有同样的尺寸。当明确为精确的尺寸后，其也就被给定了一个精确的边界
    • AT_MOST：最大值模式。子view可以一直大到指定的值。也就是其宽高属性设置为wrap_content，那么它的最大值也不会超过父布局给定的值，所以称为最大值模式

ViewGroup.LayoutParams

  • LayoutParams被view用于告诉它们的父布局它们想要怎样被布局
  • 该LayoutParams基类仅仅描述了view希望宽高有多大。对于每一个宽或者高，可以指定为以下三种值中的一个：MATCH_PARENT,WRAP_CONTENT,an exact number
  • 对ViewGroup不同的子类，也有相应的LayoutParams子类
  • 其width和height属性对应着layout_width和layout_height属性
  • 还可以margin和padding

mearsure流程

  • 布局的测量先计算最里面的View的大小，然后再根据子View的大小计算外层父View的大小，由内而外进行递归计算
  • 对于View的子类，一般需要重写onMeasure方法进行大小规则计算，而ViewGroup的onMeasure方法是个空方法，它的子类比如Framelayout或者Linearlayout必须要重写该方法

View

onMeasure(int widthMeasureSpec, int heightMeasureSpec)：根据ViewGroup的分析，我们明白了View何时调用，参数的意义

  • 根据调用方法从内向外，以width分析
    • getSuggestedMinimumWidth：根据mMinWidth和设置的背景大小取最小值
    • getDefaultSize：参数1就是最小值，参数2是根据父ViewGroup和本身大小已经算出来的MeasureSpec值，所以此时AT_MOST和EXACTLY，直接取计算好的大小即可。
    • setMeasuredDimension：调用后，此时才是真正给定了View的大小，也就是调用getMeasureWidth和getMeasureHeight才会有真正意义的值。

总结

  • 一般View的子类，可以不重写onMeaure
  • 一般ViewGroup的子类，必须重写onMeasure

Draw

View的draw 方法一般不去重写，官网文档也建议不要去重写draw 方法

注释写得比较清楚，一共分成6步

  • 绘制背景
  • 保存canvas图层
  • 绘制内容
  • 绘制子View
  • 绘制其他及保存图层
  • 绘制scrollerbar

对于第四步，绘制子View，就是遍历子View然后drawChild(),drawChild()方法实际调用的是子View.draw()方法,ViewGroup类已经为我们实现绘制子View的默认过程，这个实现基本能满足大部分需求，所以ViewGroup类的子类（LinearLayout,FrameLayout）也基本没有去重写dispatchDraw方法，我们在实现自定义控件，除非比较特别，不然一般也不需要去重写它， drawChild()的核心过程就是为子视图分配合适的cavas剪切区，剪切区的大小正是由layout过程决定的，而剪切区的位置取决于滚动值以及子视图当前的动画。设置完剪切区后就会调用子视图的draw()函数进行具体的绘制了

总结

  • Views是一个由ViewGroup包裹多级ViewGroup和View的树形结构
  • 整体绘制分成测量，布局和绘制三个部分
  • 测量是从最里面子View开始直到外层ViewGroup，一般ViewGroup的子类需要重写onMeasure该方法，View的子类也可能需要重写
  • 布局是ViewGroup对子View的位置的确定，一般ViewGroup的子类必须要重写onLayout方法，View的子类不需要重写
  • 绘制是将View的内容绘制出来，一般ViewGroup的子类不需要重写onDraw方法，View的子类大概率需要重写

View的坐标

View

  • getHeight()：获取View自身高度（measure后）
  • getWidth()：获取View自身宽度
  • getMeasureHeight():获取View自身高度(layout后)
  • getMeasureWidth():获取View自身宽度
  • getTop()：获取View自身顶边到其父布局顶边的距离
  • getLeft()：获取View自身左边到其父布局左边的距离
  • getRight()：获取View自身右边到其父布局左边的距离
  • getBottom()：获取View自身底边到其父布局顶边的距离

MotionEvent

  • getX()：获取点击事件距离父控件左边的距离，即视图坐标
  • getY()：获取点击事件距离父控件顶边的距离，即视图坐标
  • getRawX()：获取点击事件距离整个屏幕左边距离，即绝对坐标
  • getRawY()：获取点击事件距离整个屏幕顶边的的距离，即绝对坐标
